# 基于约束式执行的 AI 代码开发实践

## 一、问题的起点

在软件开发团队中，有一类任务比新建项目更难处理：**接手一个存量代码库并向其中添加新功能**。

代码库已经积累了大量历史决策和隐性约定。当新需求到来，开发者打开 AI 工具，将几个相关文件粘贴进去，让它帮忙修改。问题随即出现：

**不知道从哪里改。** 函数之间的调用关系错综复杂。AI 没有完整的代码结构上下文，给出的修改建议忽略了上下游依赖，改了一处，另一处悄然失效。

**写出来的代码格格不入。** C 语言嵌入式代码对命名规范、错误处理方式、内存管理模式有极强的项目内约定。AI 没有风格上下文，生成的代码在功能上可能正确，但风格上像是另一个人写的。

**需求无法准确传递给 AI。** 需求方的描述是业务意图，AI 需要的是可执行的技术任务。没有结构化的需求上下文，中间的翻译过程长期依赖个人经验，偏差在执行阶段以 bug 的形式重新出现。

这三个问题本质上是同一个问题：**AI 的失败从来不是能力不足，而是上下文不足。**

---

## 二、两个工具，各司其职

解决上下文问题，需要为 AI 的每一个关键决策点提供正确的上下文——在正确的时机、以正确的形式。本文介绍的实践使用了两个工具共同完成这件事：

**GSD（Get Shit Done）** 是一个开源的 AI 工作流模板，负责建立**需求上下文**：通过结构化问询将模糊意图转化为分阶段的执行计划，并在每阶段执行前与开发者对齐方案，执行后进行验证。GSD 本身是针对 CodeGraphWiki 进行适配的，它的执行引擎在生成代码前还会自动读取目标文件，建立**风格上下文**。

**CodeGraphWiki（CGW）** 是本文的核心工具，负责建立**代码上下文**：对代码库构建知识图谱与向量索引，在开发过程中精准回答「应该调用哪个现有接口、这个函数的调用者是谁、调用它之前需要满足什么前提」。

两者的分工是：GSD 构建需求侧的上下文，CGW 构建代码侧的上下文。两类上下文缺一不可——没有需求上下文，AI 做的方向可能是错的；没有代码上下文，AI 做的方式可能是错的。

---

## 三、CodeGraphWiki 如何构建代码上下文

CGW 的工作分为两个阶段：**离线建图**（构建上下文）和**运行时查询**（投递上下文）。

### 离线建图

对代码库执行一次索引，CGW 分三层构建代码上下文：

#### 第一层：提取函数与结构体

CGW 使用 Tree-sitter 对 C 语言源码进行静态分析。Tree-sitter 是一个增量解析库，直接解析源文件的抽象语法树（AST），**不需要编译、不需要链接**——这对嵌入式项目尤为重要，因为这类项目的编译往往依赖特定的交叉编译环境，在开发机上无法直接构建。

通过 AST 解析，CGW 提取：所有函数的签名与参数类型、全局和局部结构体定义、宏定义与枚举类型。提取结果存入知识图谱，每个符号成为图中的一个节点——这是代码上下文的原始材料。

#### 第二层：整理调用关系

单独的符号列表还不够。AI 在修改一个函数时，需要知道它牵动着哪些上下游依赖——这是更深一层的结构上下文。

CGW 分析每个函数体内的调用语句，构建函数调用图。图中每条有向边代表一次调用：`A → B` 表示 A 的函数体内调用了 B。对于通过函数指针发起的间接调用，CGW 会追踪指针的赋值路径，尽可能还原实际的调用目标。

调用关系图回答两类问题：

- **顺向查询**：这个函数会调用哪些下游函数？（了解实现依赖）
- **逆向查询**：这个函数被哪些地方调用？（评估修改影响范围）

#### 第三层：API 文档与 LLM 增强

前两层建立的是结构化数据，但 AI 直接消费的是语言。第三层的任务是将结构数据转化为 **agent 可直接执行的上下文**。

CGW 为每个公开函数生成结构化 API 文档，分两个来源：

**静态提取**来自代码本身：函数签名、参数类型、返回值、注释内容、调用者与依赖列表。这部分是机械的、精确的。

**LLM 增强**是新版 CGW 的核心改进。静态提取完成后，CGW 会对每个函数的文档再过一遍 LLM，生成两类内容：

- **功能描述**：用自然语言说明这个函数在系统中的作用。不是重复签名，而是解释它解决什么问题、在什么场景下被使用。
- **调用前置条件**：明确说明调用这个函数之前需要完成哪些初始化、满足哪些状态前提。这部分直接从调用关系图和函数体逻辑中推导，而不是靠人工维护。

以 `mqtt_connect` 为例，LLM 增强后的文档如下：

```
/code-graph:api-detail "mqtt_connect"

函数签名：
  int mqtt_connect(const char *host, int port, mqtt_config_t *cfg)

【功能描述】
  建立与 MQTT Broker 的 TCP 连接并完成协议握手。
  负责管理连接状态机，成功后将内部状态置为 MQTT_CONNECTED，
  后续的 publish / subscribe 调用均依赖此状态。

【调用前置条件】
  1. net_init() 必须已执行，且 net_is_ready() 返回 true
  2. 若 cfg->use_tls 为 true，需先调用 tls_init() 完成证书加载
  3. 不得在中断上下文中调用，此函数内部有阻塞等待

参数：
  host    服务器地址，不含协议前缀
  port    端口号（1883 非加密 / 8883 TLS）
  cfg     连接配置，包含 keepalive、clean_session、遗嘱消息

返回值：
  0           连接成功
  -ETIMEDOUT  连接超时（默认等待 5s，由 cfg->timeout 控制）
  -EAUTH      认证失败

调用者：system_init(), reconnect_handler()
```

「调用前置条件」代表着一类长期缺失的上下文。函数签名告诉 AI 「能调什么」，前置条件告诉它「调之前必须先做什么」。嵌入式系统中，前置条件错误是最难定位的一类 bug——函数签名完全正确，代码编译通过，但运行时以某种隐蔽的方式崩溃，原因是某个初始化步骤被遗漏。把这类隐性知识变成显式的上下文，是 LLM 增强的核心价值。

这份文档不需要手动维护。每次建图后，LLM 基于最新的代码状态重新生成，始终与代码保持同步。

### 运行时查询

建图只需执行一次，结果持久化存储，代码库发生变化时可增量更新。在执行每个开发任务前，CGW 按需将代码上下文投递给 AI，不占用主会话的上下文窗口。

## 四、api-find：精准的上下文投递

在 CGW 提供的所有能力中，`api-find` 是约束式执行流程中最关键的一步，值得单独说明。

### 它解决什么问题

传统 AI 辅助开发的上下文投递方式是「粘贴文件」：把几个看起来相关的文件复制进对话框。这种方式有两个根本性缺陷：**覆盖不完整**（漏掉了相关文件）和**精度不够**（引入了大量无关内容，稀释了有效信息）。结果是 AI 要么调用了错误的接口，要么在它「知道」的范围内自行创造实现，与现有系统割裂。

`api-find` 在代码生成之前插入一个强制步骤：**用意图查询，得到精准的上下文，再开始写**。

### 它如何工作

`api-find` 接受一个自然语言描述，在知识图谱和向量索引中同时执行查询：

- **图谱查询**：匹配功能关键词，沿调用关系找到相关节点
- **语义检索**：通过向量相似度找到功能语义接近的函数，即使名称完全不同也能命中

两路结果合并、去重、按相关性排序后返回。`api-find` 工作在三层数据之上——符号提取、调用关系图、LLM 增强文档——因此每个结果不是一条裸签名，而是一份**可直接执行的上下文**：

- **函数签名**与精确文件位置（文件路径 + 行号）
- **调用者列表**：修改这个函数时需要纳入验证的所有调用点
- **LLM 增强字段**：功能描述（这个函数做什么）+ 调用前置条件（调它之前必须满足什么）

以下是一次查询的输出示例：

```
/code-graph:api-find "MQTT connection"

→ 找到 3 个匹配接口：

mqtt_connect(const char *host, int port, mqtt_config_t *cfg) → int
  位置：src/net/mqtt.c:142
  调用者：system_init(), reconnect_handler()
  功能描述：建立与 MQTT Broker 的 TCP 连接并完成协议握手，成功后置状态为 MQTT_CONNECTED
  调用前置条件：net_init() 已执行且 net_is_ready() 为 true；不得在中断上下文调用

mqtt_publish(const char *topic, const uint8_t *payload,
             size_t len, int qos) → int
  位置：src/net/mqtt.c:287
  调用者：telemetry_send(), ota_notify_progress()
  调用前置条件：mqtt_connect() 已成功，内部状态为 MQTT_CONNECTED

mqtt_subscribe(const char *topic, mqtt_msg_handler_t handler) → int
  位置：src/net/mqtt.c:334
  调用者：system_init()
  调用前置条件：mqtt_connect() 已成功
```

对于一个正在规划 OTA 功能的 AI agent，这份结果提供了完整的行动上下文：MQTT 模块已经存在（不需要重新实现）、三个接口的精确位置（知道改哪里）、调用点有哪些（知道改了会影响什么）、调用 `mqtt_connect` 之前必须先完成网络初始化（知道怎么正确地调用它）。

### 为什么精准很重要

**调用者列表**让 AI 的上下文从「单点」扩展到「影响范围」。当 AI 知道「有两个地方调用了 `mqtt_connect`」，它在修改时会将这两个调用点纳入验证范围，而不是修改后只测试自己新加的代码。

**调用前置条件**让 AI 的上下文从「现在」延伸到「之前」。嵌入式系统中，运行时的崩溃往往来自遗漏的初始化序列，而不是错误的逻辑本身。这类知识隐藏在代码的执行顺序里，`api-find` 将它提取为显式字段，在规划阶段就交给 AI——而不是等到 debug 时才发现。

### 它在流程中的位置

```
GSD 生成执行任务
    │
    ▼
api-find "<任务关键词>"   ← 在此步获取：调用哪些接口、前置依赖是什么、影响哪些调用点
    │
    ▼
读取目标文件，建立风格上下文
    │
    ▼
在完整上下文内生成代码
```

这个顺序是强制的。CGW 的设计原则是：**没有上下文，不生成代码**。

## 五、实战流程

以为嵌入式设备添加远程固件升级（OTA）功能为例，展示完整链路中每一步上下文是如何建立和传递的。

### Step 1：建立需求上下文（GSD 问询）

执行 `/gsd:new-project` 后，GSD 通过结构化问询消除需求中的歧义：

> **GSD：** 升级包通过什么渠道下发？服务器推送还是设备主动拉取？
>
> **开发者：** 设备主动拉取，通过 MQTT。
>
> **GSD：** 升级中断后需要支持回滚吗？
>
> **开发者：** 必须支持，设备不能变砖。
>
> **GSD：** 代码库中现有 MQTT 连接管理模块吗？
>
> **开发者：** 有，OTA 逻辑没有。
>
> **GSD：** 明确了。约束：复用现有 MQTT 模块、支持中断回滚、不影响正常业务通信。生成计划。

问询结果结构化为 `REQUIREMENTS.md` 和分阶段的 `ROADMAP.md`。这是需求上下文的固化：后续所有阶段的 AI 决策，都在这个边界内进行。

### Step 2：建立代码上下文（CGW api-find）

规划「固件下载与写入」阶段时，在生成任何代码之前先执行查询：

```
/code-graph:api-find "MQTT connection"
/code-graph:api-find "Flash write"
```

两次查询将代码上下文注入规划过程：现有 MQTT 接口的完整签名、调用关系与前置条件、Flash 驱动的写入和擦除接口。规划器基于这些真实接口写出任务的执行指令，而不是抽象描述。

### Step 3：建立风格上下文 + 代码生成

执行器在生成代码前读取目标文件，建立风格上下文：观察并匹配命名规范（函数前缀、错误码定义方式）、注释格式、错误处理模式（返回值约定）。三类上下文全部到位后，AI 在已知边界内生成代码——新函数在功能上满足需求，在风格上与原文件保持一致。

## 六、总结

让 AI 真正融入存量代码库开发，核心问题不是模型能不能写出正确的代码，而是它在生成代码时手里有没有足够的上下文。

需求上下文缺失，AI 做的可能是错的事。代码上下文缺失，AI 做的方式可能破坏现有结构。风格上下文缺失，AI 的产出需要人工反复打磨才能合入。

CGW 解决代码上下文的问题：通过知识图谱和 `api-find`，在生成前精准投递「这里有什么、调用它需要先做什么」，而不是让 AI 在片段信息里猜测。GSD 解决需求上下文和风格上下文的问题：将模糊意图结构化，并在动笔前先读文件、学规范。

这套机制不依赖于某个模型版本的能力上限。上下文越充分，模型越强，产出的质量越高。反过来说，在上下文充分的前提下，即便是今天的模型，也足以完成大多数存量代码库的功能扩展任务。
